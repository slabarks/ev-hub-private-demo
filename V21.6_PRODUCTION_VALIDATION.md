## Automated result

`npm test` passed the complete eight-stage suite.

### Stage 1 — syntax and production guards

- Python compilation of `server.py`.
- JavaScript syntax for app, data, financial, technical, maturity, optimizer, export, snapshot and utility modules.
- Backend-route resilience when version metadata is unavailable or an older route returns an incomplete response.
- Deployment-manifest coverage for the coordinated engine files.
- Single navigation/readiness system and Investor/Analyst mode guards.

### Stage 2 — browser-local ZIP/XLSX parser

- Complete daily and monthly histories generated without a backend.
- Canonical file selection and Ignore-folder handling retained.

### Stage 3 — AADT regression

**18/18 tests passed**, including:

- source-data integrity and coordinate provenance;
- official-source precedence and bundled fallback disclosure;
- route-aware matching for Ballincollig, Bandon, Kinsale and M1 examples;
- fail-safe handling of coarse coordinates;
- curated mapping integrity; and
- randomized grid invariants.

### Stage 4 — Python live-history parsing

**6/6 tests passed**, including:

- complete dashboard ZIP support;
- daily-history gaps and rolling windows;
- site-specific source dates;
- supporting-workbook exclusion;
- mature and early-site monthly histories; and
- zero-demand calendar-day retention.

### Stage 5 — maturity forecasting

Passed forward-12-month separation, maturity curve, P25/P50/P75 bands, confidence handling, backtesting and 20-year forecast regression.

### Stage 6 — financial and technical integrity

Passed deterministic tests for:

- period-zero construction and year-end operating cashflow timing;
- NPV reconciliation;
- valid and invalid IRR handling;
- fractional payback;
- grant caps and non-negative investment;
- ESB application-fee propagation;
- actual installed battery initialization;
- battery SOH, reserve and dispatch fraction;
- battery service-cost reconciliation;
- downtime/reliability propagation;
- overnight recharge windows;
- secured-lease returns; and
- explicit 0% discount-rate warning.

It also passed **400 deterministic randomized scenarios** across demand, tariff, electricity cost, MIC, grant, lease, battery, reliability and actual-CAPEX boundaries. Invariants included finite outputs, no negative initial investment, grants not exceeding gross CAPEX, served energy not exceeding demand and usable battery energy not exceeding nominal energy.

Snapshot storage was also tested under simulated quota failure. Forecast calculations and successful live-data activation remain valid even when the optional browser ledger cannot persist.

### Stages 7–8 — API and static delivery

Passed:

- `/api/version` and prefixed API route;
- `/api/health`;
- AADT lookup and counter-location delivery;
- live-calibration upload through primary and prefixed routes;
- gzip/build/parser headers;
- root application delivery; and
- required static engine/parser/vendor files.

## Change-impact reconciliation

Representative V21.5/V21.6 comparisons were generated for:

- default 20-year case at 8%;
- default case at 0%;
- excessive grant support;
- installed-day-one battery; and
- actual installed project CAPEX.

The expected result is generally lower but more defensible NPV/IRR because V21.6 applies proper cashflow timing, reliability, battery reserve/dispatchability and reconciled OPEX. See `V21.6_CHANGE_IMPACT.md`.

## Manual/visual validation limitation

Automated HTML/static/API delivery completed successfully. A full Chromium visual screenshot run could not be completed in the validation environment because its browser policy blocks loopback/private-network pages, including the local test server. This is an environment restriction rather than an application error. A five-point post-deployment visual check is listed in `DEPLOY_TO_RENDER.md`.

## Release conclusion

**PASS — V21.6 is a production candidate for complete-package deployment.**

The calculation-integrity, technical-consistency, auditability, uncertainty-display and navigation changes are coordinated and should be deployed together. The items in `V21.6_COMMERCIAL_DECISIONS_PENDING.md` remain intentionally unretuned pending management approval.
